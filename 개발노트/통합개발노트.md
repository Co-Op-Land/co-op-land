## 📌 도메인 간 의존성 관리 전략

### 🧩 문제 상황
`Member`의 정보를 조회하는 `findById` 메서드는 여러 도메인에서 자주 사용된다.  
기존에는 다른 서비스 계층에서 해당 기능이 필요할 때,  
직접 `MemberRepository`를 주입하여 `memberRepository.findById(id)`와 같이 호출했다.

하지만 이 방식에는 몇 가지 문제가 있었다

---

### 🔍 문제 분석

- A 도메인에서 B 도메인의 `Repository`를 직접 참조하는 것은 👉 **불필요한 접근 권한을 부여받는 구조**이다.
- 이로 인해 **객체 간 결합도가 높아지고**, **변경에 따른 영향 범위가 커지며**, **역할 분리가 명확하지 않게 된다.**

---

## ⚖️ 서비스 vs 레포지토리 의존성 결정

도메인 간 데이터 접근 시, 어떤 계층에 의존할지에 대한 설계는 매우 중요하다.  
또한 이를 **추상화(인터페이스)**하거나, **재사용 가능한 컴포넌트**로 분리할 수 있다.

### ✅ 서비스 주입 방식

- **장점**
    - 구현이 간단하다
    - 중복 코드 제거 가능
    - 비즈니스 로직 재사용 용이
- **단점**
    - 서비스 간 결합도가 높아진다
    - 단일 책임 원칙(SRP), 의존성 역전 원칙(DIP)에 위배될 수 있다
    - 순환 참조 발생 위험

---

### ✅ 레포지토리 직접 주입 방식

- **장점**
    - 서비스 간 결합도 약화
    - 관심사 분리 명확
    - 테스트 용이
- **단점**
    - 비즈니스 로직 중복 위험 (헬퍼 메서드 등으로 보완 가능)
    - 설계 복잡성 증가

---

### ✅ 인터페이스 추상화 방식

- **장점**
    - 구현체와 추상 계층 분리 → 유연성 증가
    - DIP 원칙 준수
- **단점**
    - 코드량 증가
    - 전체 도메인에 적용할 경우 복잡도 급상승

---

### ✅ 컴포넌트 모듈화 방식

- **장점**
    - 공통 기능 재사용 가능
    - 순환 참조 문제 해결
    - 구조적으로 응집력 높은 역할 분리 가능
- **단점**
    - 코드량 약간 증가
    - 구조 통일 시 복잡도 증가 가능성

---

## 🎯 최종 선택: **조회 전용 컴포넌트 방식 (MemberReader)**

### 🔧 방식
- `MemberReader` 또는 `MemberComponent` 생성
- `findById`, `findByEmail` 등의 **조회 전용 로직만 포함**

### 📌 선택 이유
- 조회 로직은 여러 도메인에서 공통적으로 사용됨
- `MemberService`를 주입하면 불필요한 조작 메서드까지 접근 가능 → SRP 위배
- 컴포넌트를 통해 조회만 허용하는 **단방향, 저결합 구조** 설계 가능
- 생산성과 단순함 모두를 만족시킴

---

### 💡 구현 예시

```java
@Component
@RequiredArgsConstructor
public class MemberComponent {

    private final MemberRepository memberRepository;

    public Member findById(Long id) {
        return memberRepository.findById(id).orElseThrow();
    }

    public Member findByEmail(String email) {
        return memberRepository.findByEmail(email).orElseThrow();
    }

    public Member findMemberReference(Long id) {
        return memberRepository.getReferenceById(id);
    }
    //...
    }
```

## ✅ 결론
- 도메인 간 불필요한 의존을 제거하고, SRP와 DIP를 고려한 구조로 설계하여 유지보수성을 향상시킴

- 초기에는 컴포넌트로 간단히 시작하고, 시스템이 커지면 CQRS, 인터페이스 분리 등으로 점진적으로 확장 가능
