## 📌 도메인 간 의존성 관리 전략

### 🧩 문제 상황
`Member`의 정보를 조회하는 `findById` 메서드의 경우, 기존에는 타 서비스 계층에서 이 메서드를 사용하기 위해  
`MemberRepository`를 직접 의존하고 `memberRepository.findById(id)`와 같이 호출하곤 했다.

`Member` 도메인은 여러 도메인에서 공통적으로 사용되는 대표적인 도메인 중 하나이므로, 이 문제를 예시로 삼아 설명한다.

---

### 🔍 문제 분석

다른 도메인(A)이 특정 도메인(B)의 `Repository`를 직접 의존하는 구조는,  
👉 **필요 이상의 접근 권한을 갖게 된다**는 문제가 있다.  
이는 **객체 간 결합도를 높이고**, **불필요한 변경 전파의 위험**을 야기할 수 있다.

---

### 💡 해결 방안 제시

#### ✅ 기존 방식
- `MemberService`를 의존하고, 그 안에 `findById` 메서드를 만들어 사용한다.

#### 🚫 발견된 한계
- `MemberService`를 의존하게 되면 `findById` 외에도 다른 수정/삭제/조작 메서드에 접근이 가능하다.
- 이는 **서비스 간 역할 분리가 명확하지 않음**을 뜻하며, 장기적으로 유지보수에 불리할 수 있다.

---

### 🛠️ 대안 비교

| 번호 | 대안 | 장점 | 단점 |
|------|------|------|------|
| 1️⃣ | `Repository`를 조회/조작으로 분리하여 인터페이스화 | MyBatis 등 기술 확장에 유리 | 인터페이스 증가로 복잡도 상승 |
| 2️⃣ | `Service`를 인터페이스화하고 필요한 메서드만 노출 | 필요한 기능만 접근 가능 | 인터페이스 관리의 비용 증가 |
| 3️⃣ | 조회 전용 `Component`를 별도로 생성하여 사용 | 구조 단순화, 빠른 생산성 확보 | 역할 명확화 필요 |

---

### 🎯 최종 선택: 3번 - **조회 전용 Component 도입**

```java
@Component
public class MemberComponent {
    public Member findById(Long id) {
        return memberRepository.findById(id).orElseThrow();
    }
}
```
### 📝 선택 이유
- 프로젝트 초기 단계에서는 생산성과 단순한 구조가 더 중요하다.

- 인터페이스를 과도하게 도입하면 복잡도가 불필요하게 상승할 수 있다.

- 향후 확장 시 MemberReader를 기반으로 별도 인터페이스 도입이나 구조 개선이 유연하게 가능하다.

### ✅ 결론
- 도메인 간 불필요한 권한 공유를 피하고,

- 역할에 맞는 책임만 분리하여 제공하는 구조가 더 유지보수에 유리하다.

- 지금은 조회 전용 Component로 시작하고, 향후 시스템이 커질 경우 CQRS, 인터페이스 분리 등도 점진적으로 적용 가능하다.
