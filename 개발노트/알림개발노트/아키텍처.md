## 📌 프로젝트 개요

알림은 다음 두 가지 방식으로 설계하였다:

- **RDB 기반 알림**
    - 로그인 후 확인 가능한 알림 목록
    - 읽음 처리 등 상태 저장
    - WebSocket 세션이 없는 유저를 위한 Fallback

- **푸시 기반 알림 (실시간)**
    - 브라우저 푸시를 활용한 UX 중심 알림
    - 실시간 반응성 제공

---
## ⚒️️ 기술 선택 이유

### ✅ Kafka
- 알림 트래픽: 분당 최대 10만 건의 알림 처리 필요
- 원래 RabbitMQ를 고려했지만, 메시지 큐 기반으로 큐 수 증가 시 스케일링 한계 존재
- Kafka는 파티션 기반 분산 처리로 수평 확장에 강하고 **Throughput**이 높아 적합
- 실시간 외에도 RDB 저장, 재처리 등의 확장 로직에 유리
- **부하 테스트를 통해 최적의 파티션 전략 수립 예정**

---

### ✅ Redis Pub/Sub
- 실시간 브로드캐스트에 적합
- 구조가 단순하고, 딜레이가 거의 없음
- 개인적으로 Redis에 익숙해 빠르고 안정적인 구성 가능

---

### ✅ WebSocket (Spring)
- Spring에서 기본적으로 제공
- 과거에 사용해본 경험이 있어 비교적 쉽게 구현
- 익숙한 기술 스택 기반으로 빠른 개발 → 운영 및 확장도 고려된 선택

---

## ⚙️ 기본 아키텍처 구성

### 1️⃣ WebSocket 연결 (클라이언트 ↔ 서버)

- 사용자가 웹에 로그인하면, 프론트엔드에서 WebSocket 연결을 맺는다:

  ```
  ws://<서버 주소>/ws?token=Bearer <JWT>
  ```

- 해당 요청은 JwtFilter를 거치며 유효성 검증, 유저 검증을 수행하게 되고, 성공적으로 처리되면 WebSocket 세션을 **유저 ID 기준으로 메모리에 저장**한다.
- WebSocket은 연결 지향형 프로토콜로 연결이 성립되면 TCP 기반의 **고유한 채널이 생성**되고, 해당 채널은 하나의 WebSocketSession 객체로 매핑되며, 이 값을 jvm이 저장하는 것이다.
- 하나의 유저가 **여러 디바이스에서 동시 접속**할 수 있기 때문에,
  유저 ID 하나에 대해 여러 WebSocket 세션을 저장할 수 있는 구조로 설계했다.('다중 클라이언트 지원' 부분 참조)
- 추후 SSL 인증서 발급 시, wss://로 보안 연결 가능하다.

> 이 단계는 실시간 양방향 통신을 위한 기반이 된다.  
> 세션은 서버 인스턴스의 **로컬 메모리**에 저장되며, 직렬화되거나 외부에 공유되진 않는다. (직렬화되지 않는다)

---

### 2️⃣ 알림 이벤트 처리 (비동기 처리 구조)

- 예를 들어 사용자가 댓글을 작성하면, 백엔드 서버는 내부적으로 **알림 이벤트**를 발행한다.
- 알림 이벤트는 즉시 처리되지 않고, **Kafka 메시지 브로커에 발행**된다.

> Kafka는 높은 처리량과 장애 허용성을 가진 **분산 메시징 시스템**으로,  
> 알림과 같은 비동기/다중 인스턴스 환경에서의 메시지 처리에 적합하다.

---

### 3️⃣ Kafka → Redis Pub/Sub → WebSocket 흐름

#### ✅ Kafka 역할

- Kafka는 알림 메시지를 토픽(`notification.comment` 등)에 담아 브로드캐스트한다.
- 각 인스턴스마다 Kafka Consumer가 존재하며, 토픽을 구독하고 메시지를 수신한다.

#### ✅ Redis Pub/Sub 역할

- Kafka 메시지를 수신한 인스턴스는 이를 **Redis Pub/Sub 채널에 전파(Publish)** 한다.
- 모든 인스턴스는 이 채널을 구독(Subscribe)하고 있으므로, **전체 서버가 동시에 메시지를 전달받을 수 있다.**

> Kafka는 백엔드 간 통신 (비동기 전파용),  
> Redis는 WebSocket 알림 전파를 위한 **브로드캐스트 도구**로 사용.

#### ✅ WebSocket 역할

- Redis 메시지를 받은 각 인스턴스는,
  **내가 관리하고 있는 WebSocket 세션 중 이 알림의 대상 유저가 있는가**를 확인한다.
- 있다면 해당 유저에게 실시간 푸시 알림을 전송한다.

---

### 🧭 전체 구성도 요약

```
[ 사용자 (Web) ]
       ↓ WebSocket 연결 (/ws)
[ 서버 인스턴스 A ]
       ↑ 세션 등록 (유저 ID ↔ WebSocketSession)

[ 사용자 댓글 작성 ]
       ↓
[ 서버 인스턴스 A ]
       ↓ Kafka 발행 (notification.comment)
[ Kafka ]
       ↓ (구독)
[ 서버 인스턴스 B ]
       ↓ Redis 채널 Publish (notification_channel)
[ 모든 인스턴스 ]
       ↓ Redis 수신 후 세션 확인
       ↓ 대상 유저에게 WebSocket 푸시
```

---

### ✅ 핵심 설계 의도

| 구성 요소 | 역할 |
|-----------|------|
| **WebSocket** | 클라이언트에 실시간 알림 푸시 |
| **Kafka** | 서버 간 비동기 알림 이벤트 전달 (대용량 처리) |
| **Redis Pub/Sub** | 각 인스턴스에 알림 브로드캐스트 (빠름 + 경량) |
| **세션 저장소** | 인스턴스 로컬 메모리 (유저 ID → 세션 리스트) |

## 💡 다중 클라이언트 & 인스턴스 환경 개선

### ✅ 다중 클라이언트 지원

기존 구조:

```java
Map<Long, Session> sessions; // 유저당 하나의 세션만 저장
```

문제점:

- PC에서 로그인 후 모바일 접속 시, 기존 세션이 끊김

해결:

```java
Map<Long, CopyOnWriteArrayList<Session>> sessions;
```

- 불변 리스트로 여러 세션 저장 가능
- 각 기기에서 동시에 같은 알림 수신 가능

---

### ✅ 다중 인스턴스 환경 지원

#### ❌ 잘못된 접근

- WebSocket 세션을 Redis에 저장하고, 모든 인스턴스에서 가져와 쓰는 방식 시도
- 실패 원인:
    - WebSocket 세션은 직렬화 불가능
    - TCP 연결 상태는 연결된 인스턴스 메모리에만 존재

#### 🟢 해결책: Redis Pub/Sub 기반 브로드캐스트

1. WebSocket 세션은 각 인스턴스 메모리에서만 관리
2. 알림 메시지는 Redis Pub/Sub 채널을 통해 모든 인스턴스에 브로드캐스트
3. 각 인스턴스는 수신한 메시지의 대상 유저가 연결되어 있는지 확인
    - 연결되어 있으면 WebSocket으로 푸시
    - 없으면 무시

이 방식은 다음 장점을 제공한다:

- 인스턴스 간 세션 공유 없이 수평 확장 가능
- Redis 채널을 활용한 간단한 브로드캐스트 구조
- 실시간성과 확장성 동시 확보

**하지만**, 현재 '어떤 사용자가 접속 중인지' 정확하게 알 수 없기 때문에 추가 개선 필요함.  
Redis Pub/Sub은 근본적인 해결책이 아니다.  
한 유저의 세션이 여러 Jvm에 존재할 경우 중복 메세지가 발생한다.

- 해결책 ex) WebSocketSession에서 id를 추출해 <id, 최근 활동 시간> 형태로 redis에 저장
- Heartbeat 구조 (Ping-Pong) 설계 예정
