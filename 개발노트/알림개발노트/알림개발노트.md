## 1. 📌 알림 시스템 개요

실시간 알림 시스템은 사용자 경험(UX)을 향상시키고, 다양한 알림 유형을 효율적으로 처리하기 위해 설계되었다. 알림은 두 가지 방식으로 제공된다:

- **RDB 기반 알림**
    - 로그인 후 확인 가능한 알림 목록
    - 읽음 처리 등 상태 저장
    - WebSocket 세션이 없는 유저를 위한 Fallback

- **푸시 기반 알림 (실시간)**
    - WebSocket을 활용한 실시간 알림
    - 브라우저 푸시로 UX 중심의 반응성 제공

**목표**
- 다양한 알림 유형(댓글, 평가, 공지 등) 지원
- 동시 접속자 수 2-3만명, 분당 10만 건 이상의 트래픽을 안정적으로 처리
- 다중 클라이언트 및 다중 인스턴스 환경에서의 안정적 운영

## 2. ⚒️ 기술 선택 이유

### 2.1 메시지 브로커 (Message Broker)

- **선택 기술**: **Kafka**
- **이유**: Kafka는 높은 처리량(Throughput)과 수평 확장성을 제공하여 **분당 10만 건 이상의 알림 트래픽**을 안정적으로 처리할 수 있음.  
  이벤트 기반 아키텍처에 최적화된 분산 메시징 시스템으로, **RDB 저장, 실시간 전파, 처리 분기** 등 복잡한 요구에 유리함.

- **비교**:
  - **Kafka**: 파티션 기반 병렬 처리 및 로그 기반 저장 → **고속 + 영속성**
  - **RabbitMQ**: 큐 기반 브로커로 소규모에 적합. **대규모 트래픽 처리 한계**
  - **Redis Pub/Sub**: 브로드캐스트 특화. **영속성 없음**

- **추가 이점**:
  - 로그 기반 메시지 저장 → **재처리 및 장애 복원**
  - 컨슈머 그룹 활용 → **처리 로직 분산 및 병렬화**
  - 추후 **채팅 메시지까지 확장 가능**

---

### 2.2 브로드캐스트 도구 (Real-time Broadcaster)

- **선택 기술**: **Redis Pub/Sub**
- **이유**: 인메모리 기반의 **초저지연 메시지 전파**가 가능하며, 구조가 단순해 다중 인스턴스 간 알림 동기화에 적합.

- **비교**:
  - **Redis Pub/Sub**: 밀리초 단위 반응 속도, **구조 단순**
  - **Kafka Pub/Sub**: 전파 가능하나, **브로드캐스트용으로는 무거움**
  - **MQTT**: IoT 전용. 설정 복잡 + 웹 호환성 낮음
  - **RabbitMQ (fanout)**: 사용 가능하나 Redis보다 **지연 큼**

- **추가 이점**:
  - 캐시와의 통합 → **알림 상태 fallback 대응 가능**
  - 도입 용이성 → Spring Data Redis, 커뮤니티 풍부
  - 채팅 세션 캐싱 등 **추후 확장성 확보**
  - 개인적으로 Redis에 익숙해 빠르고 안정적인 구성 가능

---

### 2.3 실시간 통신 채널 (Realtime Client Channel)

- **선택 기술**: **WebSocket (Spring)**
- **이유**: **양방향 통신과 지속 연결 기반** 실시간 알림 UX를 제공하며, Spring 생태계와의 통합성이 뛰어남.

- **비교**:
  - **WebSocket**: 양방향, 지속 연결
  - **SSE**: 단방향 한계, 인터랙션 부족
  - **Long Polling**: 높은 서버 부하
  - **Kafka/WebSocket 직접 연계**: 복잡 + Redis 브릿지 필요

- **추가 이점**:
  - `wss://` 지원으로 **보안성 확보**
  - 유저 ID 기반 다중 세션 저장 가능
  - 채팅 기능 도입 시, **1:1/그룹 채팅에 바로 확장 가능**
  - 과거에 사용해본 경험이 있어 비교적 쉽게 구현 가능

---

### 2.4 기술 선택 시 종합 고려사항

#### 🧪 트래픽 특성
- **알림 기준**: 분당 10만 건 이상 / 2만 명 이상 동시 접속 예상
- **커뮤니티 특성**: 읽기 중심, 알림 지연 허용 범위 1~2초
- **미래 확장성**: 채팅 기능 도입 예정 → 실시간성 + 영속성 필요

#### ⚠️ 문제 상황
- 대규모 트래픽 처리 시 병목 가능성
- 다중 인스턴스 간 메시지 불일치 위험
- Web 기반에서 실시간 통신의 안정성 필요

#### 🧩 해결 전략
- **Kafka**: 파티션 기반 분산 처리, 병렬 컨슈머로 병목 해소 + 영속성 보장
- **Redis Pub/Sub**: 초경량 브로드캐스트, 알림 전파 속도 극대화
- **WebSocket**: 세션 기반 양방향 통신 → 실시간 UX, 채팅 대응

---

### 2.5 최종 선택: Kafka + Redis Pub/Sub + WebSocket

이 조합은 **실시간성과 확장성, 안정성을 모두 만족**하며, 알림 시스템뿐 아니라 추후 채팅 기능 도입 시에도 유연하게 대응 가능하다.

- **Kafka**: 비동기 분산 처리, 알림 및 채팅 트래픽 처리에 강함
- **Redis Pub/Sub**: 다중 인스턴스 간 초저지연 전파에 최적
- **WebSocket**: Spring 기반 브라우저 실시간 UX + 다중 세션 지원
- **통합적 장점**: 각각의 역할이 명확하게 분리되어 성능 및 유지보수 최적화
- **확장 시나리오**: Kafka 파티션 수 확장, Redis 클러스터링, WebSocket 세션 최적화 등을 통해 유연한 수평 확장 가능

## 3. ⚙️ 기본 아키텍처 구성

### 3.1 WebSocket 연결
- 사용자가 로그인하면 프론트엔드에서 WebSocket 연결을 맺는다:
  ```
  ws://<서버 주소>/ws?token=Bearer <JWT>
  ```  
- JwtFilter로 유효성 및 유저 검증 후, WebSocket 세션을 **유저 ID 기준으로 로컬 메모리에 저장**.
- WebSocket은 연결 지향형 프로토콜로 연결이 성립되면 TCP 기반의 **고유한 채널이 생성**되고, 해당 채널은 하나의 WebSocketSession 객체로 매핑되며, 이 값을 jvm이 저장하는 것이다.
- 추후 SSL 인증서 발급 시, wss://로 보안 연결 가능하다.

### 3.2 알림 이벤트 처리
- 댓글 작성 등 이벤트 발생 시, 백엔드에서 알림 이벤트를 생성해 **Kafka에 비동기 발행**.
- Kafka는 높은 처리량과 장애 허용성을 제공해 대규모 비동기 처리에 적합.

### 3.3 Kafka → Redis Pub/Sub → WebSocket 흐름

#### ✅ Kafka
- 알림 메시지를 토픽(`notification.comment`, `notification.rdb` 등)에 발행.
- 각 인스턴스의 Kafka Consumer가 토픽을 구독해 메시지 수신.

#### ✅ Redis Pub/Sub
- Kafka 메시지를 수신한 인스턴스가 Redis Pub/Sub 채널에 메시지를 Publish.
- 모든 인스턴스가 채널을 Subscribe하여 메시지를 동시에 수신.

#### ✅ WebSocket
- 각 인스턴스는 Redis 메시지의 대상 유저가 자신의 WebSocket 세션에 있는지 확인.
- 해당 유저가 있으면 실시간 푸시 알림 전송.

**구성도 요약**
```
[ 사용자 (Web) ] → WebSocket 연결 (/ws) → [ 서버 인스턴스 A ] (세션 저장)  
[ 사용자 이벤트 ] → [ 서버 인스턴스 A ] → Kafka 발행  
[ Kafka ] → [ 서버 인스턴스 B ] → Redis Publish  
[ 모든 인스턴스 ] → Redis 수신 → WebSocket 푸시
```

## 4. 🧩 구조 리팩토링: 알림 유형 확장

### 4.1 기존 문제
- 기존 시스템은 댓글 알림만 처리하도록 설계.
- 다양한 알림 유형(평가, 공지 등) 확장에 한계:
    - AOP 추출 값, 수신자 수, RDB 저장 여부 등이 알림마다 다름.

### 4.2 템플릿 메서드 패턴과의 비교
알림 유형별 로직을 처리하기 위해 **템플릿 메서드 패턴**과 **전략 패턴**을 비교했다.
- **템플릿 메서드 패턴**
  - 추상 클래스를 정의하고, 고정된 프로세스를 템플릿 메서드로 구현.
  - 하위 클래스가 특정 단계를 오버라이딩.
  - **장점**: 공통 로직을 상위클래스에 캡슐화해 코드 중복 감소.
  - **단점**:
    - 상속 구조로 인해 새로운 알림 유형 추가 시 클래스 계층이 복잡해짐.
    - 유연성이 제한적이며, 공통 흐름 외의 독립적 로직 추가가 어려움.
- **전략 패턴**
  - 인터페이스를 통해 각 알림 유형을 독립적 전략으로 구현.
  - 런타임에 전략을 동적으로 선택.
  - **장점**:
    - 새로운 알림 유형 추가 시 인터페이스 구현체만 추가하면 됨.
    - 상속 없이 독립적 로직 설계 가능, 유지보수성과 확장성 우수.
  - **단점**: 공통 로직을 각 전략에 반복 구현해야 할 가능성.  

**결론**: 템플릿 메서드 패턴은 고정된 프로세스에 적합하지만, 알림 시스템의 다양한 요구사항(수신자 처리, RDB 저장 여부 등)을 유연하게 처리하기 위해 전략 패턴을 선택했다. 공통 로직의 중복 문제는 default 메서드로 해결했다. 

### 4.3 전략 패턴 도입
- `NotificationStrategy` 인터페이스를 정의해 알림 유형별 전략 분리.
- 각 전략에서 이벤트 객체 생성 및 Kafka 전송 처리.

```java
public interface NotificationStrategy {
    NotificationEvent buildEvent(Object[] args, Object result);
}
```

**예시**:
```java
public class CommentNotificationStrategy implements NotificationStrategy {
    public NotificationEvent buildEvent(Object[] args, Object result) {
        return new CommentNotificationEvent(...);
    }
}
```

- 새로운 알림 유형 추가 시 `NotificationStrategy` 구현체만 추가.

### 4.4 공통 흐름 추상화
- 공통 로직(예외 처리, RDB 저장 등)을 default 메서드로 정의.
```java
public interface NotificationStrategy {
    NotificationEvent buildEvent(Object[] args, Object result);
    void save(NotificationEvent event);
    default void execute(Object[] args, Object result) { /* 공통 로직 */ }
}
```

**개선 예정**:
- Producer는 이벤트 생성/발행/검증만 처리.
- Consumer에서 토픽별 비동기 이벤트 처리로 Kafka의 장점(컨슈머 그룹) 활용.

### 4.5 Producer 실행 흐름
```java
@Around("@annotation(triggerNotification)")
public Object notify(ProceedingJoinPoint joinPoint, TriggerNotification triggerNotification) throws Throwable {
    Object result = joinPoint.proceed();
    Object[] args = joinPoint.getArgs();
    NotificationStrategy strategy = strategies.stream()
        .filter(s -> s.supports(triggerNotification.target()))
        .findFirst()
        .orElseThrow(() -> new InvalidRequestException(ErrorCode.INVALID_NOTIFICATION_TYPE));
    strategy.execute(args, result);
    return result;
}
```
- 각 전략을 순회하며 supports 메서드의 반환값이 true인 전략 실행(execute)

## 5. 📦 알림 RDB 저장: 일대다 관계 및 일급 컬렉션

### 5.1 배경
- 알림은 한 명 또는 여러 명의 수신자를 가질 수 있음:
    - 댓글 알림: 게시글 작성자 + 부모 댓글 작성자
    - 평가 알림: 평가 대상자 1명
    - 공지사항: 전체 유저 또는 접속 중 유저
- **1:N 관계**로 설계해 수신자별 관리(읽음 여부 등) 가능.

### 5.2 도메인 모델 설계

#### 🔹 Notification
```java
@OneToMany(mappedBy = "notification", cascade = CascadeType.ALL, orphanRemoval = true)
private List<NotificationRecipient> toMemberIds = new ArrayList<>();
```

- `cascade = ALL`로 수신자 리스트 영속화.
- `orphanRemoval = true`로 참조 제거 시 삭제.
- 전송자(`fromMemberId`), 대상(`target`, `relatedId`) 포함.

#### 🔹 NotificationRecipient
```java
@ManyToOne(fetch = FetchType.EAGER)
@JoinColumn(name = "notification_id")
private Notification notification;
```

- 수신자 정보(유저 ID, 읽음 여부 등) 관리.

### 5.3 일급 컬렉션: ToMemberIds (수신자 리스트)

**기존 문제점**: 일급 컬렉션 없이 List<Long>만 사용했을 때

1. **로직 중복**
   - 여러 클래스(전략, 서비스 등)에서 수신자 리스트에 대해 같은 로직 반복:
     - 중복 ID 제거
     - 자기 자신 제외
     - 비연결 유저 필터링  
   - → 이 로직들이 흩어져 있어서 유지보수가 어려움

2. **역할과 데이터가 분리됨**  
   - 단순 `List<Long>`은 그 자체로 의미를 가지지 못함
   - 어떤 리스트가 수신자인지, 어떤 필터가 필요한지 코드 컨텍스트 없이는 알 수 없음
   - → 도메인 모델의 의도를 드러내지 못함

3. **부작용 우려**  
   - 외부에서 직접 리스트를 조작할 수 있어, 객체 일관성 보장 어려움
   - 예를 들어 .add()나 .remove()가 아무 데서나 호출될 수 있음

4. **테스트 어려움**
   - 수신자 리스트에 대한 검증(중복 여부, 필터링 여부 등)을 테스트하려면 해당 로직이 분산돼 있어서 단위 테스트 코드가 지저분해짐

5. **도메인 규칙 강제 불가**
   - "자기 자신에게는 알림을 보내지 않는다"는 비즈니스 규칙을 개발자의 기억에만 의존하게 됨 → 실수 유발

**해결:**
- `List<Long>`를 감싸는 일급 컬렉션으로 수신자 리스트 관리.
- **역할**: 중복 제거, 자기 자신 제외, 웹소켓 세션 필터링.

```java
public class ToMemberIds {
    private final List<Long> values = new ArrayList<>();
    private Long fromMemberId;
    public void add(Long memberId) {
        if (memberId != null && !Objects.equals(memberId, fromMemberId) && !values.contains(memberId)) {
            values.add(memberId);
        }
    }
    public void addTo(Notification notification) {
        values.forEach(notification::addRecipient);
    }
}
```

**장점**:
- 도메인 용어로 가독성 향상.
- 중복 제거, 예외 처리로 응집도 증가.
- 단일 책임 원칙 준수.

### 5.4 저장 흐름
1. 전략에서 `NotificationEvent` 생성.
2. `ToMemberIds.addTo(notification)`로 수신자 등록.
3. `notificationRepository.save(notification)`로 영속화.

## 6. 🚀 서비스 플로우

### 6.1 유저 댓글 알림
```text
1. 유저 A가 댓글 작성  
2. @TriggerNotification(COMMENT) 실행  
3. CommentNotificationStrategy로 NotificationEvent 생성  
   - 대상: 게시글 작성자, 부모 댓글 작성자  
4. RDB에 Notification, NotificationRecipient 저장  
5. Kafka → Redis → WebSocket 브로드캐스트  
    - WebSocket 알림은 '현재 접속중인 유저(웹소켓 세션이 있는 유저)' 에게만 발송하여 네트워크 비용 절감.
6. 접속 중 유저는 실시간 푸시, 미접속 유저는 로그인 후 확인  
```

### 6.2 관리자 알림

#### 저장형 공지 (RDB)
```text
1. 관리자가 공지 작성 + 발송 (RDB 선택)  
2. @TriggerNotification(RDB) 실행  
3. AdminRDBNotificationStrategy로 NotificationEvent 생성  
   - 대상: Role.USER 권한 전체 유저  
4. RDB 저장 → Kafka → Redis → WebSocket  
5. 접속 중 유저는 실시간 푸시, 미접속 유저는 로그인 후 확인  
```

#### 실시간 공지 (WebSocket)
```text
1. 관리자가 공지 작성 + 발송 (WebSocket 선택)  
2. @TriggerNotification(WEBSOCKET) 실행  
3. AdminWebSocketNotificationStrategy로 NotificationEvent 생성  
   - 대상: WebSocket 연결된 유저  
4. RDB 저장 생략 → Kafka → Redis → WebSocket  
5. 접속 중 유저만 실시간 푸시 (휘발성)  
```

**관리자 알림 특징**:
- 관리자는 **전체 유저** 또는 **현재 접속 중 유저**에게 공지 형태의 알림을 보낼 수 있다.
- 관리자 API 호출 시 해당 Member가 ADMIN이 맞는지 서비스 계층에서 추가 검증하고, 로그를 남긴다.
- 관리자는 **EndPoint**(이벤트 페이지 주소, 공지사항 주소 등), **Content**(공지 내용)을 보낼 수 있다.
    - 공지 내용을 WebSocket 메세지에 포함하면, 단순 링크 전송 대비 메세지 길이 만큼 **네트워크 트래픽 비용이 추가 발생**한다.
    - 수신자 수가 많을수록, 메세지 크기가 클 수록 이 차이는 점점 커진다.
    - 따라서 URL만 전달하고 클라이언트에서 처리하는 방법도 사용할 수 있도록 설계했다.
    - EndPoint를 Null로 보낼 경우 모달로만 처리할 수도 있다.

**비교표**  

| 항목 | 유저 알림                    | RDB 공지 | WebSocket 공지 |  
|------|--------------------------|---------|---------------|  
| 대상 | 글/댓글 작성자, 평가 대상자 등       | 전체 유저 | 접속 중 유저 |  
| 저장 | O                        | O | ✕ |  
| 토픽 | `notification.comment` 등 | `notification.rdb` | `notification.websocket` |  
| 실시간 | O                        | O | O |  
| 비로그인 확인 | O                        | O | ✕ |

## 7. 💡 다중 클라이언트 및 다중 인스턴스 환경 개선

### 7.1 다중 클라이언트 지원
**문제**: 기존은 유저당 단일 세션만 저장, 다중 디바이스 접속 시 세션 끊김.  
**해결**:
```java
Map<Long, CopyOnWriteArrayList<Session>> sessions;
```
- 불변 리스트로 다중 세션 저장.
- 모든 디바이스에서 동일 알림 수신 가능.

### 7.2 다중 인스턴스 환경 지원
**잘못된 접근**: WebSocket 세션을 Redis에 저장하고, 모든 인스턴스에서 가져와 쓰는 방식 시도.
- **실패 원인**:
  - WebSocket 세션은 직렬화 불가능.
  - TCP 연결 상태는 연결된 인스턴스 메모리에만 존재.

**해결**:
- WebSocket 세션은 각 인스턴스 로컬 메모리에 저장.
- 알림 메세지는 Redis Pub/Sub으로 모든 인스턴스에 브로드캐스트.
- 각 인스턴스는 자신의 세션에 대상 유저가 있는지 확인 후 푸시.

**한계**:
- 브로드캐스트 방식은 근본적인 해결책이 아님.
- 접속 중 유저를 정확히 파악 불가, 중복 메시지 발생 가능.  
  **개선 예정**:
  - `<유저 ID, 최근 활동 시간>`을 Redis에 저장.
  - Heartbeat(Ping-Pong) 구조로 세션 관리 최적화.
