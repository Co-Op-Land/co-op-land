## 🧩 구조 리팩토링: 알림 유형 확장

### ✅ 기존 문제

- 알림 시스템은 `'댓글 알림'` 하나만을 기준으로 구현되어 있음
- 현재 구조는 다양한 알림 유형 확장에 한계  
  (AOP 추출 값, 수신자 수, RDB 저장 여부 등이 알림마다 다름)

---

### ✅ 전략 패턴 도입

- `NotificationStrategy` 인터페이스 정의
- 각 알림 유형을 전략 구현체로 분리
- 이벤트 객체 추출 및 Kafka 전송을 각 전략에서 처리

```java
public interface NotificationStrategy {
    NotificationEvent buildEvent(Object[] args, Object result);
}
```

#### 구현 예시

```java
public class CommentNotificationStrategy implements NotificationStrategy {
    public NotificationEvent buildEvent(Object[] args, Object result) {
        return new CommentNotificationEvent(...);
    }
}
```

- Producer는 전략을 찾아 해당 이벤트를 Kafka에 전송
- 새로운 전략 추가 시 `NotificationStrategy` 구현만 하면 됨

---

### ✅ 예외 처리, RDB 저장 등 공통 흐름 추상화

#### 공통 흐름은 default 메서드로 처리

```java
public interface NotificationStrategy {

    // AOP 이용하는 로직
    NotificationEvent buildEvent(Object[] args, Object result);

    // RDB 저장 로직, 실시간 알림(ex 게임 매칭 성공) 인 경우 오버라이딩만 수행
    void save(NotificationEvent event);
    
    default void execute(Object[] args, Object result) {
        // 전체 로직 수행 기본 로직
    }
}
```
추후 개선 사항: Producer에서는 이벤트 구조 생성 + 발행 + validate만 처리하고,
Consumer에서 알림 비동기 이벤트 기반 분기 처리.   
현재 구조는 Kafka를 단순 메세지 전달 수단으로만 사용하고 있어,   
토픽별 분리 및 컨슈머 그룹 기반의 비동기 이벤트 처리 구조라는 Kafka의 장점을 활용하지 못하고 있음.

---

### ✅ Producer의 실행 흐름

```java
@Around("@annotation(triggerNotification)")
public Object notify(ProceedingJoinPoint joinPoint, TriggerNotification triggerNotification) throws Throwable {
    Object result = joinPoint.proceed();
    Object[] args = joinPoint.getArgs();

    // supports 메서드에서 return값이 true인 strategy 탐색
    NotificationStrategy strategy = strategies.stream()
            .filter(s -> s.supports(triggerNotification.target()))
            .findFirst()
            .orElseThrow(() -> new InvalidRequestException(ErrorCode.INVALID_NOTIFICATION_TYPE));

    // 어노테이션의 target 변수에 따라 알맞은 전략 로직 실행(kafka로 메세지 전송)
    strategy.execute(args, result);
    return result;
}
```

## 📦 알림 RDB 저장을 위한 일대다 관계 구현과 일급 컬렉션 적용

### ✉️ 배경

알림 시스템에서 수신자는 한 명일 수도 있고, 여러 명일 수도 있다.  
예를 들어:

- 댓글 알림 → 게시글 작성자 + 부모 댓글 작성자
- 평가 알림 → 평가 대상자 1명
- 관리자 공지사항 → 전체 유저, 또는 현재 접속중인 유저

따라서 하나의 알림(Notification)이 여러 수신자(NotificationRecipient)를 가질 수 있는 **1:N 관계**가 필요하다.  
이를 효율적으로 설계하고 유지보수성을 높이기 위해, **일급 컬렉션**을 함께 활용했다.

---

### 🧩 도메인 모델 설계

### 🔹 Notification

```java
@OneToMany(mappedBy = "notification", cascade = CascadeType.ALL, orphanRemoval = true)
private List<NotificationRecipient> toMemberIds = new ArrayList<>();
```

- 하나의 Notification은 여러 명의 수신자를 가질 수 있다.
- `cascade = ALL`로 함께 영속화되고, `orphanRemoval = true`로 참조 제거 시 삭제됨
- 알림 전송자(`fromMemberId`)와 대상(`target`, `relatedId`) 포함

```java
public void addRecipient(Long toMemberId) {
    NotificationRecipient recipient = NotificationRecipient.builder()
        .notification(this)
        .toMemberId(toMemberId)
        .build();
    this.toMemberIds.add(recipient);
}
```

### 🔹 NotificationRecipient

```java
@ManyToOne(fetch = FetchType.EAGER)
@JoinColumn(name = "notification_id")
private Notification notification;
```

- 알림 수신자 정보를 별도로 관리 (유저 ID, 읽음 여부 등)
- `@ManyToOne`으로 Notification과 연결

---

### 🧠 일급 컬렉션: `ToMemberIds`

`List<Long>` 기반의 수신자 ID 리스트를 감싸는 **일급 컬렉션**으로 다음 역할을 수행한다:

- 중복 제거 및 자기 자신 제외
- `Notification` 객체에 수신자 리스트를 일괄 추가
- 현재 웹소켓 세션을 가진 유저만 필터링

```java
public class ToMemberIds {
    private final List<Long> values = new ArrayList<>();
    private Long fromMemberId;

    public void add(Long memberId) {
        if (memberId != null && !Objects.equals(memberId, fromMemberId) && !values.contains(memberId)) {
            values.add(memberId);
        }
    }

    public void addTo(Notification notification) {
        values.forEach(notification::addRecipient);
    }
}
```

### ✅ 장점

- **도메인 용어로 의미를 부여**하여 가독성 증가
- 중복 제거, 예외 처리 등 **로직의 응집도 증가**
- **단일 책임 원칙 준수**

---

### ⚙️ 저장 흐름 예시

1. 알림 전략에서 `NotificationEvent`를 구성
2. `ToMemberIds.addTo(notification)` 호출로 수신자 등록
3. `notificationRepository.save(notification)` 호출 시 recipient 리스트도 함께 영속화됨

---

## ✅ 정리

- 다대일 관계 대신 **1:N 구조**로 알림 수신자 개별 관리
- 읽음 처리, 필터링 등의 로직 분리 용이
- `ToMemberIds` 일급 컬렉션을 통해 수신자 리스트를 안전하게 처리

> 알림 시스템에서 수신자 정보는 중요한 도메인 요소다.  
> 일급 컬렉션은 단순한 데이터 리스트 이상의 의미를 부여하며, 도메인 주도 설계의 좋은 실천 예시가 된다.
