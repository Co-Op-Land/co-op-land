## 🧩 코드 플로우 (공통 구조)

```text
@TriggerNotification 어노테이션
       ↓
NotificationProducer (AOP)
- 메서드 실행 후 result 획득
- NotificationTarget 기반 Strategy 선택(supports 메서드)
       ↓
NotificationStrategy (interface)
- buildEvent() → 알림 이벤트 생성
- validate() → 필터링
- save() → RDB 저장
- publish() → Kafka 전송
       ↓
KafkaTemplate → "notification.<type>" 토픽
       ↓
Kafka Consumer 수신 → Redis Pub/Sub 브로드캐스트
       ↓
WebSocketSessionManager (각 인스턴스)
- 세션에 대상 유저가 있으면 WebSocket 전송
- WebSocket 통신 비용 절감을 위해 현재 접속 중인 유저에게만 푸시를 수신한다. 
```
현재 Consumer에서 Consumer Group에 따라 분기 처리 없이 단순 Redis Pub/Sub 브로드캐스트만 수행하고 있는데,   
추후 재설계 예정.

## 🚀 서비스 플로우: 유저 댓글 알림

```text
1. 유저 A가 게시글에 댓글 작성
   ↓
2. 댓글 생성 메서드에 @TriggerNotification(COMMENT) 적용
   ↓
3. AOP에서 COMMENT 전략(CommentNotificationStrategy) 실행
   ↓
4. 글 ID, 작성자 정보로 NotificationEvent 생성
   - 알림 수신 대상: 게시글 작성자, 부모 댓글 작성자
   ↓
5. RDB에 Notification, NotificationRecipient 저장
   ↓
6. Kafka → Redis → WebSocket 브로드캐스트
   ↓
7. 접속 중 유저는 실시간 푸시 수신
   - 미접속 유저는 로그인 후 알림 목록에서 확인 가능
```

## 🛡️ 서비스 플로우: 관리자 알림

- 관리자는 **전체 유저** 또는 **현재 접속 중 유저**에게
공지 형태의 알림을 보낼 수 있다.
- 관리자는 EndPoint(이벤트 페이지 주소, 공지사항 주소 등), Content(공지 내용)을 보낼 수 있다.
  - 공지 내용을 WebSocket 메세지에 포함하면, 단순 링크 전송 대비 메세지 길이 만큼 네트워크 트래픽 비용이 추가 발생한다.
  - 수신자 수가 많을수록, 메세지 크기가 클 수록 이 차이는 점점 커진다.
  - 따라서 URL만 전달하고 클라이언트에서 처리하는 방법도 사용할 수 있도록 설계했다.
  - EndPoint를 Null로 보낼 경우 모달로만 처리할 수도 있다.

### 저장형 공지

```text
1. 관리자가 공지 작성 + 발송 (RDB 방식 선택)
   ↓
2. @TriggerNotification(RDB) 실행
   ↓
3. AdminRDBNotificationStrategy 선택
   - 알림 수신 대상: Role.USER 권한 전체 유저
   ↓
4. NotificationEvent 생성 → RDB 저장
   ↓
5. Kafka → Redis → WebSocket
   ↓
6. 접속 중 유저는 실시간 푸시 수신
   - 미접속 유저는 로그인 후 알림 목록에서 확인 가능
```

- `MemberComponent.getMemberIdsByRole(Role.USER)`로 대상 조회
- RDB 저장으로 읽음 처리 가능

---

### 실시간 공지

```text
1. 관리자가 공지 작성 + 발송 (WebSocket 방식 선택)
   ↓
2. @TriggerNotification(WEBSOCKET) 실행
   ↓
3. AdminWebSocketNotificationStrategy 선택
   - 알림 수신 대상: 현재 WebSocket 연결된 유저 목록
   ↓
4. NotificationEvent 생성 → RDB 저장 생략
   ↓
5. Kafka → Redis → WebSocket
   ↓
6. 접속 중 유저는 실시간 푸시 수신
   - 휘발성 알림, 저장 X
```

- `WebSocketSessionManager.getConnectedUserIdList()` 활용
- 로그인 안 한 유저는 해당 알림을 받을 수 없음

## ✅ 정리

| 항목 | 일반 유저 알림                 | 관리자 RDB 공지         | 관리자 WebSocket 공지 |
|------|--------------------------|--------------------|------------------------|
| 대상 유저 | 글 작성자, 부모 댓글 작성자 등       | 전체 유저 (DB)         | 현재 접속 중 유저 |
| 저장 여부 | O                        | O                  | ✕ |
| Kafka 토픽 | `notification.comment 등` | `notification.rdb` | `notification.websocket` |
| 실시간 전송 | O (WebSocket)            | O                  | O |
| 비로그인 시 확인 가능 | O (RDB)                  | O (RDB)            | ✕ |
| 목적 | UX 피드백 + 보관              | 유저 참여도 확인 (읽음 여부)  | 실시간 공지 |

---

> 새로운 알림 유형이 추가될 경우 `NotificationStrategy`만 확장하면 된다.  
> 이 문서는 Consumer Group 별 리팩토링 이후 한번 더 수정 예정임.
