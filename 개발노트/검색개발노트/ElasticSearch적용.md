## 1. ElasticSearch 도입 배경

### 1.1 기존 방식(MySQL-Fulltext 인덱스 + n-gram)의 한계

- 기존에는 MySQL의 Fulltext 인덱스와 n-gram 방식을 활용하여 유사 검색을 구현했으나 확장성 문제에 직면함
- **주요 문제점**
    - 데이터가 100만 건을 초과하면서 검색 응답 시간이 기하급수적으로 증가
    - 검색 처리 시 CPU 사용률 급상승으로 다른 서비스에까지 영향
    - 복합 키워드 검색 시 정확도 저하 및 검색 누락 현상 빈번 발생
    - 검색어 자동완성 및 랭킹 조정 기능 구현의 어려움
- **시스템 영향**
    - 피크 시간대 검색 지연으로 사용자 이탈률 증가
    - DB 서버 부하 증가로 전체 시스템 성능 저하

### 1.2 프로젝트 요구사항

- **성능 요구사항**
    - 대용량 데이터(100만 건 이상)에서도 1s 이내 응답 속도
    - 동시 검색 요청 처리 능력(초당 100회 이상)
- **기능 요구사항**
    - 검색어 자동완성 및 추천 기능
    - 다양한 필터(카테고리, 날짜, 가격 등) 조합 검색 지원
    - 검색 결과 관련성 및 우선순위 조정 가능
    - 한글/영어 혼합 검색 및 초성 검색 지원
- **관리 요구사항**
    - 검색 통계 및 로그 분석 용이성
    - 시스템 확장 용이성

### 1.3 결론

ElasticSearch는 위 요구사항을 모두 충족할 수 있는 검색 특화 솔루션으로, 
확장성과 유연성을 고려했을 때 최적의 선택이라고 판단함. 
특히 한글 검색에 특화된 nori 분석기와 다양한 검색 기능을 활용하여 사용자 경험을 크게 개선할 수 있을 것으로 기대됨.

## 2. 아키텍처 설계: 데이터 저장 방식에 대한 고찰

### 2.1 ElasticSearch에 모든 데이터 저장

- **장점**
    - 검색 성능 극대화
    - 데이터 변환 오버헤드 없음
    - 시스템 아키텍처 단순화
- **단점**
    - 관계형 데이터 모델링의 한계
    - 트랜잭션 관리 미지원
    - 데이터 무결성 보장의 어려움
    - 장애 발생 시 데이터 복구 복잡성

### 2.2 ElasticSearch와 DB에 모두 저장 (데이터 복제)

- **장점**
    - 각 시스템의 강점 활용 가능
    - 시스템 장애 시 상호 백업 역할
- **단점**
    - 데이터 정합성 유지 복잡성
    - 스토리지 비용 증가
    - 동기화 로직으로 인한 개발 복잡도 증가
    - 양쪽 모두 업데이트 필요한 오버헤드

### 2.3 DB에 데이터 저장, ES는 검색용 인덱스만 유지

- **장점**
    - DB의 트랜잭션 및 데이터 무결성 보장
    - 명확한 책임 분리
    - 각 시스템의 최적 용도로 활용
- **단점**
    - 실시간 검색 결과 반영에 지연 가능성
    - 동기화 메커니즘 필요
    - 시스템 복잡성 증가

### 2.4 접근 방식별 장단점 비교

| 검색 성능   | ⭐⭐⭐⭐⭐  | ⭐⭐⭐⭐  | ⭐⭐⭐⭐           |

### 2.5 데이터 정합성과 일관성 관리

**문제 상황**
- ES는 기본적으로 트랜잭션을 지원하지 않아 DB와의 동기화 시 불일치 발생 가능성 존재
- 실시간성과 정합성 사이의 균형점 찾기 필요

**원인 분석**
- DB 트랜잭션 성공 후 ES 인덱싱 실패 시 검색 결과에서 누락
- 네트워크 지연으로 인한 동기화 타이밍 불일치
- 대량 데이터 처리 시 ES 색인 과부하로 인한 시스템 전체 성능 저하

**해결 전략**
1. 이벤트 기반 아키텍처 도입 (Kafka/RabbitMQ)
2. 비동기 처리와 재시도 메커니즘 구현
3. 정합성 검증 배치 작업 주기적 실행

**현재 트래픽 특성**
- 동시접속자 20,000명
- 분당 게시글 생성 약 200건 (초당 3~4건)
- 읽기 위주 커뮤니티 특성
- 색인 지연 5~10초 허용 가능

**결론**
- Kafka나 RabbitMQ는 오버엔지니어링일 수 있음
- 내부 비동기 큐 기반 처리로 충분히 대응 가능

**구현 전략**
1. DB 저장 후 색인 작업을 내부 큐에 등록 (`ConcurrentQueue`, `ThreadPoolExecutor` 등 활용)
2. 별도 비동기 작업자가 큐에서 데이터를 가져와 Elasticsearch에 색인 처리
3. 일정 건 이상일 경우 Bulk API로 묶어서 색인 처리
4. 실패 건은 로그 저장 후 정합성 검증 배치로 후처리

**향후 확장 고려**
- 트래픽 증가 추세나 시스템 복잡도 증가 시 Kafka로 전환 가능한 구조 설계 고려 (loosely coupled)

### 2.6 최종 선택: DB + ES 하이브리드 방식

검토 결과, DB를 원천 데이터 저장소로 활용하고 ES는 검색 전용 인덱스로 운영하는 방식이 최적의 선택으로 판단됨. 이 방식은 다음과 같은 이점을 제공:

- DB의 트랜잭션 안정성과 데이터 무결성 활용
- 검색 성능 최적화를 위한 ES의 특화 기능 활용
- 시스템 간 책임 분리로 유지보수성 향상
- 각 시스템에 최적화된 쿼리 활용 가능

### 2.7 ES와 DB의 동기화 전략

#### 비동기 처리 + Bulk API

- **구현 방식**
    - 데이터 변경 이벤트를 내부 비동기 큐에 발행
    - 소비자가 이벤트를 일정 시간/개수만큼 배치로 수집
    - ElasticSearch Bulk API를 통해 배치 처리

- **최적화 포인트**
    - 초기 실시간 동기화에서 발생한 ES 과부하 문제를 해결하기 위해 배치 사이즈 조정
    - 기본 20건 단위로 처리하되, 시스템 부하에 따라 동적 조정
    - 처리 큐를 분산하여 병렬 처리 구현

#### 스케줄러 기반 정합성 검증

- **구현 방식**
    - 정기적으로(1시간 간격) DB와 ES 데이터 비교
    - 불일치 데이터 식별 및 복구 작업 수행
    - 동기화 실패 이력 모니터링 및 알림

- **장애 대응**
    - 재처리 큐를 통한 실패 항목 자동 재시도
    - 임계치 초과 시 관리자 알림
    - 유실 데이터에 대한 전체 재동기화 체계 구축

## 3. 시스템 동작 흐름

### 3.1 검색 요청 처리 흐름

```
사용자 검색 요청 → API Gateway → 검색 서비스 
→ ElasticSearch 질의 → 결과 가공 → 응답 반환
```

- **최적화 포인트**
    - 검색 쿼리 캐싱으로 반복 요청 처리 속도 개선
    - 검색 결과 페이지네이션 최적화
    - 필터 조합별 성능 모니터링 및 튜닝

### 3.2 데이터 저장 및 동기화 흐름

```
데이터 등록/수정 → RDBMS 트랜잭션 처리 → 이벤트 발행(Kafka) 
→ 이벤트 소비자 처리 → 배치 수집 → ElasticSearch 인덱싱
```

- **장애 대응**
    - 각 단계별 실패 처리 및 로깅
    - 비동기 처리의 완료 상태 추적
    - 배치 처리 실패 시 자동 재시도 로직

## 4. 한글 검색 최적화를 위한 nori 분석기 적용

### 4.1 형태소 분석기 설정

Elasticsearch의 기본 분석기는 한글 검색에 최적화되어 있지 않아, 한국어 형태소 분석에 특화된 nori 플러그인을 적용

### 4.2 사용자 사전 및 불용어 필터 적용

- **문제점**
    - 일반 형태소 분석으로는 특정 도메인 용어나 신조어 분석 정확도 부족
    - '한국어', '한국의', '한국적인' 등 유사 표현의 토큰화 일관성 부족

- **개선 방안**
    - 사용자 정의 사전 적용으로 도메인 특화 용어 추가
    - 불용어 처리로 검색 노이즈 감소
    - 동의어 사전 구축으로 검색 결과 풍부성 확보

## 5. 검색어 자동완성 기능 구현

### 5.1 Edge n-gram
#### 동작 원리
- Edge N-gram은 입력 문자열을 지정된 길이(n-gram)로 분할하여 인덱싱하는 방식
예를 들어, "스마트폰"이라는 단어는 min_gram=2, max_gram=10 설정 시 다음과 같이 분할됨

  - ```스마, 스마트, 스마트폰```

- 사용자가 입력한 검색어(예: "스마")를 동일한 분석기로 처리하여 인덱스와 매칭
- ElasticSearch의 text 필드에 edge_ngram_tokenizer를 적용하여 검색어의 접두어를 기반으로 결과를 반환
- 검색 결과는 일반 검색 쿼리처럼 유연하게 필터링 및 정렬 가능

#### 장점
- 유연성: 일반 텍스트 검색과 동일한 쿼리 메커니즘을 사용하므로 필터링, 정렬, 가중치 적용이 용이
- 다양한 검색 패턴 지원: 접두어 외에도 중간 문자열 매칭 가능(예: "마트"로 "스마트폰" 검색)
- 커스터마이징 용이: Nori 분석기 등과 결합하여 한글 검색 최적화 가능

#### 단점
- 속도: Completion Suggester에 비해 상대적으로 느림(일반 검색 쿼리 처리 방식 사용)
- 인덱스 크기: N-gram 토큰 생성으로 인덱스 크기가 증가(예: "스마트폰" 단어 하나가 여러 토큰으로 저장)
- 복잡한 설정: 적절한 min_gram과 max_gram 설정 필요(너무 짧으면 노이즈, 너무 길면 비효율적)

### 5.2 Completion Suggester
#### 동작 원리
- Completion Suggester는 ElasticSearch에서 자동완성 전용으로 설계된 고성능 기능
- 입력 문자열을 접두어 기반으로 빠르게 매칭하며, Finite State Transducer(FST) 데이터 구조를 활용해 메모리 내에서 빠른 검색을 수행
- completion 필드 타입을 사용하여 인덱싱하며, 각 문서에 제안어와 가중치를 저장
- 예: "스마트폰"을 제안어로 저장 시, "스" 또는 "스마" 입력으로 즉시 매칭

#### 장점
- 속도: FST 기반으로 매우 빠른 응답 시간(50ms 이내)
- 메모리 효율: 인덱싱된 제안어는 압축된 형태로 저장
- 간단한 쿼리: 전용 API로 구현 간소화

#### 단점
- 유연성 제한: 접두어 매칭만 지원(중간 문자열 매칭 불가)
- 가중치 관리 복잡: 동적 가중치 업데이트 시 재인덱싱 필요
- 한글 처리 제한: Nori 분석기와의 통합이 제한적(별도 전처리 필요)
- 메모리 사용량: 대량의 제안어 저장 시 메모리 소모 증가

### 5.3 Edge n-gram vs Completion Suggester

두 가지 방식을 비교 검토

| 기능      | Edge n-gram | Completion Suggester |
|---------|-------------|----------------------|
| 속도      | 중간          | 매우 빠름                |
| 구현 복잡도  | 낮음          | 중간                   |
| 메모리 사용량 | 중간          | 높음                   |
| 유연성     | 높음          | 제한적                  |
| 가중치 적용  | 용이          | 부분 지원                |

### 5.4 최종 선택
Edge n-gram 방식 채택
- 한글 검색에서 Nori 분석기와의 통합 필요
- 복합 키워드(예: "스마트폰 케이스") 및 중간 문자열 매칭 지원
- 동적 가중치 조정이 용이하여 인기 검색어 관리 효율적
